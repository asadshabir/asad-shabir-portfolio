## Response snapshot

Diagnosed three issues via OODA loop: (1) `.env.local` had `VITE_API_URL` pointing to HF Spaces URL with literal `\n` appended, breaking both estimator and reviewer services; (2) chatbot frontend had a 30-second timeout that was too short for the LLM fallback chain (Groq 8b → Groq 70b → Gemini) which can take 40-60s; (3) no delay between model fallbacks exacerbated rate limit issues. Fixed `.env.local` to use `/api` (local proxy), increased chatbot timeout to 70s, and added 1.0s `asyncio.sleep()` between model fallback attempts.

## Outcome

- ✅ Impact: Three services now function correctly end-to-end through the Vite proxy. `.env.local` fix ensures estimator and reviewer call the local backend instead of a malformed HF Spaces URL. Chatbot timeout extension prevents premature aborts during slow LLM fallbacks. Backend sleep delay prevents hammering rate limiters.
- 🧪 Tests: Manual curl tests confirmed estimator works (200), reviewer works (200), chatbot multi-turn works through turn 4 (previously failed at turn 3). Timing improved from 51s failure to ~48s success with 70s timeout.
- 📁 Files: 3 files modified
- 🔁 Next prompts: Monitor Groq API free tier token limits (100K/day), consider upgrading API keys or adding token budgeting.
- 🧠 Reflection: The root cause was subtle — `.env.local` generated by Vercel CLI had a `\n` at the end of the API URL, which would be invisible in most editors but completely broke fetch requests. The chatbot timeout was the second critical issue: the free tier Groq API ratelimits after a few requests, forcing the full fallback chain which exceeds the default 30s frontend timeout.
